# 使用指南

这份文档说明部署完成后，应该如何理解 `A / B / C / 路由` 这些概念，以及如何在 UI 里完成常见操作。

## 四个核心概念

### A：Clash 订阅

这是你的节点来源池。项目会从这里加载、测试和筛选候选节点。

### B：上游 Socks5 落地

这是最终出口的上游代理，通常是住宅代理或其他你已经持有的 Socks5 资源。

### C：服务器对外导出

这是客户端真正连接的地址，也就是 UI 最终导出的 `socks5://user:pass@host:port`。

### 路由

一条路由等于一套独立运行的组合：

- 选中的 A 节点
- 当前 B 落地
- 当前 C 监听口和认证信息

多条路由可以并行存在，并且可以同时运行。

## 推荐首次使用流程

1. 登录 WebUI
2. 保存 `A` 订阅
3. 刷新订阅并查看节点状态
4. 进入落地页录入 `B`
5. 启动当前路由
6. 到导出页测试 `C`
7. 复制导出的 `socks5://` 链接给客户端使用

## 首页看板在看什么

首页主要做两件事：

- 帮你快速确认当前所选路由是否正常运行
- 把 A、B、C 三层状态汇总在一起，方便判断故障大致在哪一层

适合在这些时候先看首页：

- 部署后第一次确认服务是否正常
- 启动路由后做快速检查
- 怀疑某条线路不稳定时做总体判断

## A 订阅管理怎么用

`A 订阅管理`页主要负责：

- 保存订阅 URL
- 刷新订阅
- 加载节点
- 测试单个节点
- 批量测试节点
- 把当前选中的 A 节点应用到当前路由

推荐做法：

1. 先保存订阅 URL
2. 点击刷新订阅
3. 载入 A 节点
4. 先做一轮筛选和测速
5. 再把合适的 A 节点应用到当前路由

## B 落地管理怎么用

`B 落地管理`页主要负责：

- 创建路由
- 复制路由
- 删除路由
- 保存当前路由的 B 落地信息
- 启动或停止当前路由
- 测试当前 B

适合的典型操作：

- 一条 A 订阅，对接多个不同国家或地区的 B 落地
- 一条已调好的路由，复制一份再换新的 B
- 批量筛查哪些 B 当前可用

## C 导出管理怎么用

`C 输出页`负责：

- 保存当前路由的 C 参数
- 查看最终导出的链接
- 测试当前 C 是否能用
- 做公网导出侧的诊断

你在这里最终会拿到标准格式：

```text
socks5://user:pass@host:port
```

这就是给客户端使用的最终链接。

## 多路由推荐用法

如果你同时维护多条线路，推荐这样组织：

- 按地区建路由，例如“印尼落地 01”“美国落地 02”
- 先把共用 A 订阅刷新好
- 再分别为不同路由录入不同 B
- 每条路由分别启动、分别测试

这样做的好处是：

- 不同 B 不会互相覆盖
- 不同 C 监听口可以并行使用
- 你可以逐条停启，而不影响其他路由

## 什么时候测 A、什么时候测 B、什么时候测 C

### 测 A

适合判断：

- 订阅里的候选节点是否还活着
- 哪个 A 节点延迟更低

### 测 B

适合判断：

- 当前上游 Socks5 是否连得通
- 当前落地资源是否还有效

### 测 C

适合判断：

- 最终给客户端的导出是否真的可用
- 外网视角是否能通过这条链路走通

## 常见操作建议

### 场景一：只跑一条线

最简单做法：

1. 用默认路由
2. 录入一条 B
3. 选一个稳定的 A
4. 启动路由
5. 复制导出链接

### 场景二：同一个 A，配多条 B

推荐做法：

1. 先把 A 订阅刷新好
2. 复制当前路由
3. 给每条新路由改不同 B
4. 分别启动
5. 到导出页分别复制不同 C 链接

### 场景三：排障

推荐顺序：

1. 先测 A
2. 再测 B
3. 最后测 C

这样更容易定位到底是订阅节点问题、落地问题，还是最终导出问题。

## 日志和诊断

UI 里会给出当前状态摘要，但如果你在系统层排障，常用入口有：

```bash
sudo journalctl -u clash-socks-webui -n 200 --no-pager
sudo systemctl status clash-socks-webui --no-pager -l
sudo ss -lntp
```

更详细的运行结构说明可以继续看 [架构说明](ARCHITECTURE.md)。
