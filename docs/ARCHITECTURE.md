# 架构说明

这份文档解释项目是怎么组织运行时组件的，以及为什么要这样设计。

## 运行时组件

### 1. FastAPI WebUI

负责：

- 登录
- Web 页面渲染
- API 接口
- 配置保存
- 路由启停和测试

入口文件：

- `app/main.py`

### 2. 路由与运行时管理层

负责：

- 生成多路由配置
- 维护 `mihomo` 进程
- 测试 A、B、C 三层状态
- 生成最终导出链接
- 恢复已跟踪的运行中路由
- 清理残留进程

核心文件：

- `app/gateway_multi.py`

### 3. mihomo

这是实际承载代理链路的核心二进制。项目会尝试自动下载合适的 Linux `amd64` 版本，并在每条路由启动时生成对应配置。

### 4. systemd

负责让 WebUI 在服务器重启后自动恢复，并提供标准化的启动、停止和日志入口。

### 5. nginx

负责处理公网入口。默认设计是：

- `nginx` 对外监听
- `uvicorn` 只监听本机

这样做比直接把 `uvicorn` 暴露到公网更稳，也更容易后续接 HTTPS。

## 默认端口关系

典型默认值如下：

```text
公网用户 -> nginx :18080 / :80 -> uvicorn 127.0.0.1:18081
客户端 -> socks5://host:10808 -> mihomo -> A -> B -> 目标网站
```

其中：

- `18080` 是默认 WebUI 公网入口
- `80` 是可选的简化入口
- `18081` 是应用内网监听口
- `10808-10999` 是默认 C 端口池

## 为什么是多路由设计

项目不是把所有配置都塞进一个全局实例，而是按“路由”组织：

- 每条路由有自己的 B、C、controller 信息
- 每条路由可以独立启动和停止
- 一条路由出问题时不一定影响另一条
- 更适合维护多个国家、多个出口、多个业务场景

## A、B、C 为什么要分层测试

很多代理问题如果只看“最终能不能连”，定位会很慢。

分层测试的意义是：

- A 测试告诉你订阅节点池是不是还健康
- B 测试告诉你上游落地是否可用
- C 测试告诉你最终导出给客户端的结果是否正常

这样比直接盲猜“哪里坏了”更容易排障。

## 运行数据存放在哪里

项目运行时会使用这些目录：

- `data/`
  保存业务配置、路由配置、provider 文件等

- `runtime/`
  保存 `mihomo` 二进制和版本信息

- `logs/`
  保存运行日志

## 为什么要在启动时清理残留进程

如果之前的 `mihomo` 子进程意外脱离管理，可能出现：

- WebUI 以为已经停了，但端口其实还在监听
- 重启后出现旧实例和新实例混在一起
- 面板状态与实际进程状态不一致

因此当前逻辑会在启动时：

- 尝试清理残留的失管 `mihomo`
- 恢复仍然应该运行的已跟踪路由

## 为什么默认不公开 docs

这个项目是运维型 WebUI，不是给外部开发者暴露 API 的 SaaS 服务。

因此默认关闭：

- `/docs`
- `/openapi.json`

这样可以减少额外暴露面，也避免把生产环境调试接口直接挂在公网。

## 为什么默认要求公网 Socks5 认证

如果一个公网 Socks5 监听不要求用户名密码，就很容易变成匿名公开代理入口。

因此当前逻辑要求：

- 只要不是本机回环监听
- 就必须有成对的 `gateway username / password`

这可以显著降低误开放的风险。
